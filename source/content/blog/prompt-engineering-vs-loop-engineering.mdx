Durante años, la habilidad más cotizada en el ecosistema de inteligencia artificial fue saber escribir buenos prompts. Cursos, certificaciones, roles completos — todo giraba alrededor de una premisa: si encuentras las palabras exactas, el modelo produce resultados mágicos. Pero en junio de 2026, algo cambió. El foco se movió del prompt al loop. Y no es un matiz menor: es un cambio de paradigma en cómo construimos con LLMs.

El término "loop engineering" fue empujado a la corriente principal por figuras como Peter Steinberger (creador de OpenClaw), Addy Osmani (Director en Google Cloud) y Boris Cherny (líder de Claude Code en Anthropic). Steinberger lo resumió con claridad en un post viral: "deberías dejar de hacer prompting a agentes de código y empezar a diseñar loops que hagan prompting a esos agentes". La frase resonó porque describía exactamente lo que ya estaban haciendo los equipos de ingeniería más avanzados.

## ¿Qué es realmente el loop engineering?

Loop engineering es la práctica de diseñar sistemas de feedback autónomo donde un LLM no responde una sola vez, sino que entra en un ciclo continuo de acción, observación, evaluación y corrección. No se trata de escribir un prompt perfecto, sino de construir el entorno de ejecución que permite al modelo iterar hasta alcanzar un objetivo verificable.

Un loop típico tiene cuatro fases fundamentales:

1. **Entrada y objetivo**: el sistema recibe una meta en lenguaje natural ("arregla los tests que fallan en este repositorio", "revisa el inventario y genera órdenes de reposición")
2. **Generación y acción**: el modelo produce un plan y ejecuta herramientas — leer archivos, llamar APIs, modificar código, ejecutar comandos
3. **Verificación y feedback**: el sistema comprueba automáticamente los resultados — tests, compiladores, logs, estado de APIs
4. **Decisión e iteración**: si la verificación pasa, avanza. Si falla, el error se inyecta de vuelta al contexto y el modelo corrige y reintenta

El bucle se repite hasta que se alcanza una condición de salida verificable o se agota un presupuesto predefinido. Como señala el artículo de Knightli (10 junio 2026), "el valor del loop está en automatizar el 'mira el resultado, encuentra el error, inténtalo de nuevo'". El humano deja de ser un intermediario que copia errores y reescribe prompts manualmente en cada iteración.

## Prompt Engineering vs Loop Engineering: las diferencias clave

La comparación no es "uno es mejor que el otro", sino "cada uno resuelve un tipo de problema distinto". El Blockchain Council publicó recientemente una guía detallada que desglosa las diferencias:

**Unidad de diseño.** El prompt engineering diseña la entrada a una llamada al modelo. El loop engineering diseña el patrón operativo completo alrededor de múltiples llamadas, incluyendo herramientas, validación y control de flujo.

**Modelo de control.** En prompt engineering, el humano dirige: tú decides qué preguntar a continuación. En loop engineering, el sistema dirige: el workflow decide la siguiente acción basándose en observaciones. Puede continuar, ramificarse, reintentar o detenerse. Esto lo acerca más a la arquitectura de software que a la redacción.

**Manejo de errores.** Un prompt engineering maneja errores mejorando las instrucciones: añadir ejemplos, ajustar restricciones, pedir otro formato. Un loop engineering maneja errores mediante tests y flujo de control: si el modelo devuelve JSON inválido, el loop valida contra un esquema, solicita reparación y falla tras dos intentos. Si una API falla, el loop captura el error, registra la petición incorrecta y corrige la configuración del llamante.

**Habilidades requeridas.** El prompt engineer necesita escritura clara, conocimiento del comportamiento del modelo y dominio del contexto. El loop engineer necesita todo eso más pensamiento sistémico: APIs, colas, reintentos, estado, logging, control de versiones, automatización de tests y límites de seguridad.

Como lo resume TechTalks (22 junio 2026): "los loops amplifican la habilidad de ingeniería, no la reemplazan".

| Aspecto | Prompt Engineering | Loop Engineering |

| Foco | Calidad de una entrada/salida | Calidad de ciclos repetidos de acción |

## El linaje de los agentic loops

El loop engineering no surgió de la nada. Tiene un linaje claro que TechTalks y promptengineering.org trazan con precisión:

- **2022**: los patrones ReAct (Reasoning + Acting) introducen ciclos de razonamiento-acción que permiten a los modelos interactuar con herramientas externas.
- **2023**: AutoGPT y proyectos open-source experimentan con agentes autónomos que se auto-dirigen.
- **2025**: aparece el "Ralph loop", scripts bash de una línea que envuelven llamadas a LLMs en bucles simples. Andrej Karpathy acuña "vibe coding" — el desarrollador "vibea" con el modelo en lugar de escribir sintaxis.
- **2026**: plataformas como Claude Code y Codex integran comandos `/goal` y `/loop`. El loop engineering madura como disciplina con orquestación jerárquica, sub-agentes especializados y persistencia de estado.

Boris Cherny, creador de Claude Code, lo describió en su charla en Meta's @Scale como "el salto que define 2026 — tan grande como el paso de código fuente a agentes".

## Cuándo usar cada enfoque

No todo requiere un loop. El prompt engineering sigue siendo la herramienta correcta para tareas acotadas: resumir un documento, generar contenido, clasificar tickets de soporte, traducir textos. Si la tarea es puntual, verificable en una sola mirada humana y no requiere interacción con sistemas externos, un buen prompt es suficiente.

El loop engineering encaja cuando la tarea es multi-paso, involucra herramientas externas y necesita verificación automatizada. Los casos de uso más claros son:

**Agentes de código.** Un agente lee un issue, inspecciona archivos, edita código, ejecuta tests, lee los fallos y corrige. Como señala el artículo de Blockchain Council, "un loop bien diseñado puede inspeccionar un codebase, cambiar archivos, ejecutar tests, leer el fallo, hacer una corrección más pequeña y parar cuando la tarea está completa o es insegura".

**Automatización empresarial.** Excepciones de facturación, verificaciones de compliance, onboarding de clientes, triaje de tickets IT. El paso de IA interpreta lenguaje; el loop controla enrutamiento, reintentos, logs de auditoría y puertas de aprobación.

**Investigación y análisis.** Un sistema loop-engineered no se limita a preguntar "resume las tendencias en NVMe". Usa MCP (Model Context Protocol) para buscar en ArXiv, obtener papers, extraer afirmaciones contradictorias y luego pedir al modelo que resuelva esas contradicciones. El loop no para hasta que la puntuación de resolución supera un umbral.

## Los riesgos: loopmaxxing y deuda de comprensión

Con la adopción acelerada vienen riesgos reales. TechTalks advierte contra el "loopmaxxing": asumir que ejecutar un agente en un ciclo infinito eventualmente producirá una solución correcta. Esto falla estrepitosamente con objetivos subjetivos o no cuantificables. Pedirle a un loop que "mejore la experiencia de usuario de esta página de login" o "genere una estrategia de marketing viral" lo deja sin condición de salida concreta. El loop corre indefinidamente, convirtiendo presupuesto de infraestructura en facturas de API.

Incluso en entornos deterministas como la ingeniería de software, los loops no supervisados encuentran mínimos locales. Karpathy admitió que en AutoResearch, los agentes a menudo se volvían conservadores: en lugar de explorar rediseños arquitectónicos audaces, el loop oscilaba modificando una tasa de aprendizaje en fracciones de punto porcentual durante docenas de ciclos.

Además, Addy Osmani advierte sobre la "deuda de comprensión": cuando un loop automatizado modifica cientos de líneas sin supervisión humana, la velocidad de generación de código supera la capacidad del equipo para revisarlo. Los desarrolladores heredan un codebase donde las decisiones de diseño, dependencias estructurales y casos límite están completamente sin mapear.

## Construir loops con cabeza: un enfoque por fases

TechTalks y promptengineering.org proponen una aproximación progresiva:

1. **Observación con humano en el loop.** Construir el harness inicial con herramientas de contexto, pero exigir aprobación humana explícita para cada modificación. Esta fase expone dónde falla la lógica del agente.

2. **Salidas deterministas.** Reemplazar revisiones humanas con validación automatizada binaria donde sea posible: compiladores, linters, suites de tests. La condición de salida debe apoyarse en checks deterministas, no en pedirle al LLM que evalúe su propia salida.

3. **Circuit breakers de estancamiento.** Añadir código de monitoreo que detecte si el agente alterna entre estados idénticos, repite errores de consola o no muestra progreso medible en tres ciclos consecutivos. Si se detecta, el loop debe cortar y alertar a un ingeniero.

4. **Destilación y degradación.** Analizar logs para identificar comportamientos repetitivos predecibles. Si el agente usa el LLM consistentemente para tareas de parsing o refactorización estructural idénticas, extraer esos pasos del prompt no determinista y reescribirlos como bloques de script compilado.

## El futuro: el prompt no muere, se integra

Prompt engineering no desaparece. Se convierte en un componente dentro del sistema de loop. Como señala Knightli, "sigues necesitando objetivos claros, restricciones y formatos de salida, pero la verificación, el estado, los reintentos y las condiciones de parada deciden si las tareas de larga duración realmente tienen éxito".

La distinción que propone promptengineering.org es elegante: el prompt le dice al modelo cómo pensar en esta ronda; la herramienta define qué puede hacer; el estado registra dónde está el proceso; el test decide si el resultado es fiable; el stop hook decide cuándo debe parar el sistema.

Para quien empieza en IA generativa, el prompt engineering sigue siendo el punto de entrada natural: enseña cómo responden los modelos a instrucciones, ejemplos, ambigüedad y restricciones. Pero para quien ya construye software, el camino es directo hacia el loop engineering: tool calling, orquestación de agentes, workflows test-driven, integración CI y monitoreo.

La pregunta ya no es "¿qué prompt debería usar?" sino "¿qué debería permitírsele hacer al agente, cómo lo verificamos y cuándo debe parar?".
